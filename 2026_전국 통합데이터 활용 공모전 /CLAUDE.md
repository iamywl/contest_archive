# 2026 전국 통합데이터 활용 공모전 — 작업 지침

이 저장소는 **행정안전부·한국지역정보개발원 주관 2026 전국 통합데이터 활용 공모전** 출품(책크인, Check-In to Read)을 위한 작업 공간이다. 모든 기여자는 아래 원칙과 산출물 규격을 엄격히 따른다.

---

## 디렉터리 규칙

```
/
├── CLAUDE.md          # (이 문서) 작업 지침
├── docs/              # 문서용 산출물 — 모두 Markdown
│   ├── 제안서.md
│   ├── 개발계획서.md
│   ├── 개발보고서.md
│   ├── 검증보고서.md
│   ├── 설문조사.md
│   └── screenshots/
├── dev/               # 개발용 산출물
│   └── chekcin/       # Next.js 데모 앱
├── todo/              # 사용자 수행 체크리스트 (API키·배포·녹화 등)
└── (공고문 원본 hwpx)
```

- **문서용(`docs/`)과 개발용(`dev/`)을 섞지 않는다.**
- 새 문서는 반드시 Markdown(`.md`)으로 작성한다. hwpx/PDF 변환은 최종 제출 단계에서만.

## 문서별 규격

### 1) `docs/제안서.md`

- **목적**: 공모전 심사용 [붙임4] 서비스 개발 기획서의 본문. 공고 지정 6개 섹션 순서 준수.
- **핵심 요구사항**
  - **참고문헌(References)을 반드시 포함**한다.
  - 정부·공공·학술·언론 출처 우선. 출처별 URL/발행기관/발행연도 명시.
  - 본문의 수치·주장은 각주 또는 괄호 인용으로 출처 번호와 연결 (예: `(KOSIS, 2024)` 또는 `[1]`).
  - 10페이지 내외 분량, 팀원 개인정보는 포함하지 않는다.
- **섹션 순서(공고 고정)**
  1. 개발 결과물 개요
  2. 개발 결과물의 기능 및 특징
  3. 기존 서비스와 차별성 및 독창성
  4. 공공데이터 등 데이터 활용성
  5. 개발 결과물의 기술성과 구현 및 실현가능성
  6. 기대 효과
- **마지막**: `## 참고문헌` 섹션 필수.

### 2) `docs/개발계획서.md`

- **목적**: 개발 일정·상태를 팀 내부와 심사위원에게 한눈에 보여주는 문서.
- **핵심 요구사항**
  - **표 위주**로 기술. 장문 서술 최소화.
  - **기술 스택 표** 포함 (레이어·기술·선정사유).
  - **개발 일정 표**: 언제부터(Start) ~ 언제까지(End) + **현재 상태(Status)** 열 포함. 상태값은 `✅ 완료 / 🟡 진행중 / ⬜ 예정 / ⚠️ 지연`.
  - **현재 진척도 요약**(% 또는 완료/전체)을 문서 상단에 배치.
  - 마일스톤·리스크·대응도 표로 정리.

### 3) `docs/개발보고서.md`

- **목적**: 실제 구현 결과물의 동작을 증거와 함께 보고하는 문서.
- **핵심 요구사항**
  - **실제 구현된 소프트웨어의 화면을 캡처**하여 첨부. 캡처는 `docs/screenshots/` 에 저장, Markdown 이미지 링크로 삽입.
  - **실제 구동 후 캡처**한다 (모의 목업 금지). 구동 환경(포트·브라우저·해상도) 명시.
  - 캡처를 **직접 검토**하여 결함·어색한 UI·의도와 다른 부분을 식별.
  - **미흡하거나 의도와 다르면 즉시 수정** → 재캡처 → 반영. 수정 이력 기록.
  - 각 캡처 아래: "무엇을 보여주는가 / 의도 / 검토 결과 / 조치" 형식의 설명.

## 코드 원칙

- `dev/chekcin/` 내 Next.js 앱은 프로덕션 빌드(`pnpm build`)가 항상 통과해야 한다.
- 데모 안정성을 위해 공공 API 키가 없을 때도 mock fallback으로 동작해야 한다.
- UI 변경 시 반드시 `pnpm dev` 실행 → 스크린샷 캡처 → 개발보고서에 반영.
- 스크린샷 캡처 자동화 스크립트는 `dev/chekcin/scripts/` 하위에 둔다.

## 코드 표준 (언어별)

린터·포맷터 설정은 이 표준에 맞춰 둔다.

### TypeScript / JavaScript (`dev/chekcin/`)

| 범주 | 규칙 |
|---|---|
| 런타임 | Node.js 20 LTS 이상 |
| 모듈 | ESM(`"type": "module"` 또는 `.mjs/.ts`), CommonJS 지양 |
| 스타일 | Prettier 기본값 + 프로젝트 ESLint (`eslint.config.mjs`) |
| 네이밍 | 변수·함수 `camelCase`, 컴포넌트·타입·클래스 `PascalCase`, 상수 `UPPER_SNAKE`, 파일 `kebab-case` 또는 컴포넌트만 `PascalCase.tsx` |
| 타입 | `any` 금지, `unknown` 후 좁히기. public API는 명시적 타입 |
| Import | 절대 경로(`@/...`) 우선, 순환 참조 금지, 사용하지 않는 import 즉시 제거 |
| React | 함수형 컴포넌트만. Hook 의존성 배열 정확히 기재 |
| 비동기 | `async/await` 우선, Promise 체이닝 지양. 에러는 경계에서 처리 |
| 주석 | WHY만. WHAT은 타입·이름으로 표현 |
| 테스트 | `*.test.ts(x)` 동위치, AAA 구조 |

### Python (참고용 — 이 프로젝트에선 미사용이지만 스크립트 추가 시 적용)

| 범주 | 규칙 |
|---|---|
| 스타일 | PEP 8 (line length 100), `ruff` |
| Docstring | PEP 257 |
| 타입 힌트 | PEP 484 — public 함수·메서드는 타입 명시 |
| 네이밍 | 함수·변수 `snake_case`, 클래스 `PascalCase`, 상수 `UPPER_SNAKE` |
| 예외 | bare `except:` 금지 |

### Markdown

| 범주 | 규칙 |
|---|---|
| 표준 | CommonMark + GFM (표·체크박스) |
| 제목 | `#` 문서당 1개, 이하 계층 구조 유지 |
| 코드블록 | 언어 태그 필수 (` ```ts `, ` ```bash ` 등) |
| 링크 | 상대 경로, 한국어 텍스트 링크 허용 |

### Shell

| 범주 | 규칙 |
|---|---|
| Shebang | `#!/usr/bin/env bash` |
| 안전 모드 | `set -euo pipefail` |
| 변수 | `"$var"` 인용 |
| 린트 | `shellcheck` 통과 |

## 글쓰기 원칙

- 한국어로 작성. 용어는 공고문 표기를 따른다.
- 배점이 높은 항목(참신성 30, 공공데이터 활용성 25, 국민편익 20)에 살을 붙인다.
- AI 가점(+10) 요건은 2군데 이상 명시한다.
- 근거 없는 수치·단정은 쓰지 않는다. 수치를 쓰면 반드시 출처를 명기한다.
