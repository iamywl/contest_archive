# [붙임5] 공모전 설문조사

**2026년 전국 통합데이터 활용 공모전**

---

## 1. 전국 통합데이터 7종을 활용하는 과정에서 어려웠던 점

### 가. 데이터 구조 및 메타데이터

- **기관별 데이터 스키마 표준화 부족**: 같은 "공공도서관 열람실 현황"이어도 지자체·
  기관별로 필드명(`seatCnt` vs `total_seat` vs `seatTotal`)과 좌석 분류 체계가
  상이하여, 서비스 측에서 공통 모델로 정규화하는 전처리 비용이 컸음.
- **코드값·분류체계 설명 부재**: 열람실 유형(일반/노트북/디지털/유아 등)을 나타내는
  코드값이 기관별로 다르고 공식 코드북이 제공되지 않아, 현장 확인 또는 경험적
  매핑에 의존해야 했음.

### 나. API 사용성

- **실시간 API 응답 지연 편차**: 기관에 따라 응답 시간이 수 초~수십 초까지 편차가
  커, 다수 도서관 일괄 조회 시 클라이언트 UX가 저하됨. 내부적으로 **5분 TTL 캐시
  + stale-while-revalidate** 전략으로 우회.
- **Rate Limit 정책의 불투명**: 개발 초기 테스트 중 간헐적 429 오류 발생. 정책이
  명확히 안내되지 않아 안전 호출 주기를 경험적으로 설정해야 했음.
- **CORS·인증 방식 혼재**: REST와 SOAP, 서비스키 위치(Query/Header) 등이 혼재되어
  통합 클라이언트 개발 난이도 상승.

### 다. 데이터 간 연계·결합

- **공간 키(위경도) 정밀도 편차**: 일부 도서관·보관함 데이터의 위경도 소수점 자리가
  달라 지도 마커 위치가 어긋나는 사례 발견.
- **시계열 일관성 부재**: 좌석 데이터는 5분 주기 스냅샷만 제공될 뿐, 과거 시계열은
  별도로 축적해야 함 → AI 혼잡도 예측 모델 학습용 데이터를 서비스 측에서 직접
  쌓아야 하는 부담.

### 라. 샘플·예제

- **실사용 시나리오 기반 샘플 부족**: 단일 API 응답 예시는 제공되나, **"도서관 +
  자전거 + 보관함"처럼 여러 API를 결합해 서비스화하는 레퍼런스 코드·튜토리얼**이
  부족했음.

## 2. 활용하신 데이터에 추가로 필요한 정보

| 추가 필요 항목 | 필요 이유 |
|---|---|
| **좌석 유형 상세 분류(콘센트 유무, 노트북 가능 여부, 스터디룸 구분)** | 이용자별 니즈에 맞춘 세분화 추천. "콘센트 있는 노트북석만 보기" 등의 필터 구현 필요. |
| **운영 시간·휴관일 정보 표준화 필드** | 실시간 좌석이 0인 이유가 "만석"인지 "휴관"인지 구분 불가. 잘못된 이동 안내를 방지하려면 운영 상태 flag 필요. |
| **과거 시간대별 평균 점유율** | AI 혼잡도 예측 모델의 학습 초기 정확도 확보. 과거 1개월 시간대별 평균만이라도 개방되면 서비스 초기 품질이 크게 향상됨. |
| **공영자전거 거치대 고장·점검 상태** | "이용 가능 대수 > 0"이지만 실제 현장 고장률이 존재. 고장 플래그가 제공되면 이용자 허탕을 줄일 수 있음. |
| **보관함 크기·보관 가능 품목 정보** | 백팩·캐리어 보관 가능 여부 필터링. |
| **혼잡도 혹은 대기 인원의 원천 데이터 표준화** | 현재는 "이용 중 좌석 수"만 있어 대기열·예약 대기를 구분하기 어려움. |
| **데이터 항목별 상세 정의서(한글 메타데이터)** | 필드명만으로는 의미 파악이 어려워, 기관별 매뉴얼 혼재로 통합 개발 지연. |
| **전국 통합 API 게이트웨이(단일 엔드포인트)** | 기관별 URL·키 관리의 표준화. 이용자(개발자)가 단일 Base URL + 기관 ID로 호출하도록 일원화되면 활용이 획기적으로 늘어날 것. |

## 3. 신규로 개방되기를 희망하는 공공데이터

### 제안 ①. **공공도서관 좌석 예약·대기열 실시간 정보**

- **활용 목적**: 이동 전 예약을 통해 허탕 최소화. 현재 "실시간 잔여좌석"만으로는
  "도착했을 때도 자리 있을까?"라는 불확실성을 해소하지 못함.
- **기대 효과**: 도서관 좌석 충돌·무단 점유 감소, 이용 만족도 상승, 도서관 운영
  효율 향상.

### 제안 ②. **공공도서관 과거 시간대별 점유율 통계 (최근 1년 이상)**

- **활용 목적**: AI 혼잡도 예측 모델 학습 데이터. 현재 실시간 API만으로는 초기
  서비스가 체크인 로그를 스스로 축적할 때까지 예측 품질이 제한됨.
- **기대 효과**: 데이터 기반 학습 루틴 추천의 정확도 대폭 향상, 공공 AI 서비스의
  초기 진입 장벽 해소.

### 제안 ③. **공공도서관 전기 콘센트·노트북석·스터디룸 상세 배치 정보**

- **활용 목적**: 사용자 니즈별 좌석 추천. "콘센트 있는 자리만"이라는 카공족의
  핵심 요구에 대응.
- **기대 효과**: 공공도서관이 스터디카페를 실질적으로 대체할 수 있는 정보 인프라
  확보.

### 제안 ④. **지역별 스터디카페·독서실 분포 및 요금 정보 (민관 협업)**

- **활용 목적**: 민간 부가 서비스와의 가격·접근성 비교로 이용자가 합리적 선택을
  하도록 지원.
- **기대 효과**: 시장 투명성 제고, 저소득 학습자의 대안 탐색 용이.

### 제안 ⑤. **공영자전거·공유킥보드·버스·지하철 멀티모달 경로 통합 API**

- **활용 목적**: "자전거 대여 → 공공도서관 도보 → 보관함"처럼 여러 교통·시설을
  체인으로 엮는 경로 최적화. 현재는 각 데이터 소스가 분리되어 있어 서비스 측에서
  자체 매칭 로직을 구현해야 함.
- **기대 효과**: 교통·생활 SOC의 진정한 통합 활용, MaaS(Mobility as a Service)
  공공 버전 구현 가능.

## 4. 공모전 참가 및 데이터 활용에 대한 기타 의견

1. **"전국 통합데이터"라는 브랜딩의 실체화가 필요합니다.**
   현재 7종 데이터는 같은 공공데이터포털에 게시되어 있을 뿐, 호출 방식·인증·응답
   스키마가 기관별로 다릅니다. 단일 **전국 통합 API 게이트웨이**와 **통합 SDK(JS/
   Python)** 가 제공되면, 본 공모전 이후에도 다양한 서비스가 폭발적으로 생겨날
   것입니다.

2. **실시간 공공데이터와 AI의 결합을 제도적으로 장려해 주세요.**
   본 공모전에서도 AI 활용에 10점 가점이 부여되지만, **실시간 데이터 × 예측
   모델**은 단순 챗봇보다 더 큰 공공 가치를 창출합니다. 과거 시계열 통계, 학습용
   샘플 데이터셋의 **상시 개방**이 이 방향을 가속화할 수 있습니다.

3. **데모 수준 결과물 평가 시 '실제 동작'을 최우선해 주세요.**
   상용 완성도보다 "입력→처리→출력 플로우의 신뢰성"과 "실데이터 기반 결과 재현성"
   이 심사 지표로 확립되면, 참가자들이 형식적 화면 구성보다 **본질적 문제 해결**에
   집중할 수 있습니다.

4. **수상작 오픈소스화 지원 프로그램**
   수상작을 공공재로 활용하기 위한 오픈소스 전환·지자체 확산 지원 프로그램이
   제도화되면, 공모전의 사회적 파급 효과가 1회성이 아닌 지속적 자산으로 남을
   것입니다.

5. **공모 일정 및 접수 방식**
   이메일 접수 방식은 간명하나, **제출 확인·수정·문의 트래킹**을 위한 간단한
   웹 포털이 제공되면 참가자 경험이 개선될 것입니다.
