## 2. 문서 작성 규칙 (docs/)

### 2.1 공통 규칙
- **언어**: 한국어 (전문 용어는 원문 병기)
- **포맷**: Markdown (확장자 `.md`)
- **다이어그램**: Mermaid (GitHub/VSCode 자동 렌더)
- **이모지**: 섹션 식별용으로만 사용 (남용 금지)
- **링크**: 상대 경로 (`[보고서](./개발보고서.md)`)

### 2.2 (1) 제안서.md 작성 규칙

#### A. 참고문헌은 **반드시** 달 것
- 모든 통계·인용·법적 주장에 `[^번호]` 각주 필수
- 각주는 문서 말미 §16 근거자료 섹션에 통합 정리
- **각주 형식 표준**:
  ```markdown
  [^1]: **기관명 「자료명」** (발표연월). 핵심 수치 인용. URL.
  ```
- 1차 출처 우선: KOSIS·통계청·고용노동부·안전보건공단·법령
- 학술 논문은 저자·연도·학술지 명기
- URL은 가능한 모두 명시 (검증 가능성)

#### B. 참고문헌 조사 우선순위
1. **공공 통계 포털** (KOSIS, e-나라지표, 공공데이터포털)
2. **정부 부처 발표자료** (고용노동부·법무부·통계청)
3. **공공기관 보고서** (한국노동연구원·산업안전보건공단·이민정책연구원)
4. **학술 논문** (KCI·DBpia·RISS)
5. **국제 기구** (ILO·OECD·UN — 비교 자료)
6. **NGO 보고서** (이주노동자지원단체·국가인권위원회)

#### C. 데이터 정직성 의무
- 검증 안 된 수치는 **`[추정]`** 명시
- 본문 마지막에 **데이터 정직성 선언** 필수
- 추정값과 공식 수치 명확히 구분

### 2.3 (2) 개발계획서.md 작성 규칙

#### A. 표 위주 작성 (서술 최소화)
- 모든 일정·기술스택·진척은 **마크다운 표**로 표현
- 1개 표는 최대 12행 이내, 초과 시 분할

#### B. 필수 섹션
| 섹션 | 내용 |
|---|---|
| 1. 기술 스택 | 계층별 표 (선택·버전·근거) |
| 2. 일정 (Gantt) | 시작일·종료일·현재상태 표 + Mermaid Gantt |
| 3. 마일스톤 | 날짜·산출물·달성여부 표 |
| 4. 스프린트 진척 | 스프린트별 상태 (✅ 완료 / 🚧 진행 / ⏸ 대기) |
| 5. 현재 상황 (Status) | 매 업데이트 시 갱신 |
| 6. 위험·이슈 | 발생일·영향·대응 표 |
| 7. 자원 사용 | API 비용·인프라 비용 표 |

#### C. 현재 상황 갱신 의무
- 코드 변경 / 빌드 / 배포가 있을 때마다 §5 현재 상황 표 갱신
- 갱신 시 `last_updated: YYYY-MM-DD HH:MM` 헤더 갱신

### 2.4 (3) 개발보고서.md 작성 규칙

#### A. 캡처 의무
- **실제 구동된 소프트웨어 화면을 반드시 스크린샷**으로 캡처
- 캡처 파일은 `docs/captures/` 에 저장 (PNG 권장)
- 마크다운에 임베드: `![설명](./captures/파일명.png)`

#### B. 작성 절차 (반드시 이 순서)
1. **구동**: 코드 빌드/실행 → 실제 화면 띄움
2. **캡처**: 핵심 화면 스크린샷 (홈·핵심 기능·결과화면 최소 5장+)
3. **작성**: 캡처와 함께 보고서 본문 작성
4. **검토**: 캡처가 의도와 일치하는지 점검
5. **수정**: 미흡하면 코드 수정 → 재구동 → 재캡처 → 보고서 재작성
6. **반복**: 의도와 일치할 때까지

#### C. 검토 체크리스트 (보고서 마지막에 포함)
- [ ] 모든 핵심 기능이 캡처되었는가
- [ ] 캡처가 의도한 기능을 정확히 보여주는가
- [ ] 한글이 깨지지 않는가
- [ ] 에러 화면이 의도치 않게 캡처되지 않았는가
- [ ] 결과물(PDF·텍스트)의 정확도가 충분한가

#### D. 보고서 구조
| 섹션 | 내용 |
|---|---|
| 1. 구현 범위 | 무엇을 만들었는가 (스코프 표) |
| 2. 환경 | OS·언어·라이브러리 버전 표 |
| 3. 실행 방법 | 누구나 재현 가능한 단계 |
| 4. 화면 캡처 | 캡처 + 설명 (각 화면당 1단락) |
| 5. 검증 결과 | 의도대로 작동하는지, 정확도 측정값 |
| 6. 미흡 사항 | 솔직하게 기술 + 다음 액션 |
| 7. 검토 체크리스트 | 위 C 항목 |

## 3. 개발 규칙 (src/)

### 3.1 언어·도구
- **Python**: 3.11+ (3.14 사용 가능)
- **패키지 관리**: `pyproject.toml` + `pip` (또는 `uv`)
- **포맷터**: `ruff` (line length 100)
- **타입**: 가능한 type hint 명시
- **테스트**: `pytest`

### 3.2 코드 스타일
- 함수·변수: snake_case
- 클래스: PascalCase
- 상수: UPPER_SNAKE
- docstring: 한국어 OK, 1줄 요약 + 필요 시 상세
- 주석: WHY만 (WHAT은 코드로)

### 3.3 환경변수 / 시크릿
- API 키는 **절대 코드에 하드코딩 금지**
- `.env` 파일 사용, `.gitignore` 등록
- 키 부재 시 **모킹 모드**로 동작하게 설계 (데모 가능성 보장)

### 3.4 의존성
필요 시에만 추가, 사용하지 않는 의존성은 즉시 제거.

기본 스택:
- `streamlit` — 웹 데모 UI
- `openai` — Whisper STT
- `anthropic` — Claude LLM
- `reportlab` — PDF 생성
- `python-dotenv` — 환경변수
- `pydantic` — 데이터 검증

## 4. 작업 흐름 (Workflow)

매 세션 또는 큰 작업 단위마다:

```
1. CLAUDE.md 재확인
2. TodoWrite 로 작업 계획
3. 코드/문서 변경
4. 코드 변경 시 → 실제 구동 → 캡처
5. docs/개발계획서.md §5 현재 상황 갱신
6. 의도와 다르면 → 수정 → 재구동 → 재캡처
7. docs/개발보고서.md 갱신 (캡처 포함)
8. TodoWrite 완료 처리
```

## 5. 절대 하지 말 것

- ❌ 출처 없는 통계 인용
- ❌ 추정값을 사실로 표현
- ❌ 캡처 없이 "구현 완료" 보고
- ❌ API 키 하드코딩
- ❌ 검증 안 된 라이브러리 무분별 추가
- ❌ docs/ 외부에 문서 산재
