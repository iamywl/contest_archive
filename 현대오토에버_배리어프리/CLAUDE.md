```

├── 공고문+2026+...pdf            ← 콘테스트 공고 원본
│
├── docs/                        ← 문서용 (제안서·계획서·보고서)
│   ├── 제안서.md
│   ├── 개발계획서.md
│   ├── 개발보고서.md
│   ├── 근거자료_조사.md
│   ├── captures/                ← 개발보고서용 스크린샷
│   └── 참고_제안서_v1.html
│
└── src/                         ← 개발용 (실제 구현 코드)
    ├── ai-pipeline/             ← Killer Feature: STT → LLM → PDF
    │   ├── app.py               ← Streamlit 웹 데모 (캡처 대상)
    │   ├── stt.py               ← Whisper 호출
    │   ├── form_mapper.py       ← 양식 매핑 (RAG)
    │   ├── pdf_renderer.py      ← PDF 생성
    │   ├── templates/           ← 공식 양식 (산재신청서·임금체불 진정서)

    └── docker-compose.yml

## 2. 문서 작성 규칙 (docs/)

### 2.1 공통
- **언어**: 한국어 (전문 용어는 원문 병기)
- **포맷**: Markdown (`.md`)
- **다이어그램**: Mermaid
- **이모지**: 섹션 식별용으로만 사용, 남용 금지
- **링크**: 상대 경로 사용

### 2.2 제안서.md

**A. 참고문헌 필수**
- 모든 통계·인용·법적 주장에 `[^번호]` 각주 필수
- 각주는 문서 말미 §16 근거자료 섹션에 통합 정리
- 각주 형식:
  ```markdown
  [^1]: **기관명 「자료명」** (발표연월). 핵심 수치 인용. URL.
  ```

**B. 조사 우선순위**
1. 공공 통계 포털 (KOSIS, e-나라지표, 공공데이터포털)
2. 정부 부처 발표자료 (고용노동부·법무부·통계청)
3. 공공기관 보고서 (한국노동연구원·산업안전보건공단·이민정책연구원)
4. 학술 논문 (KCI·DBpia·RISS)
5. 국제 기구 (ILO·OECD·UN)
6. NGO 보고서 (이주노동자지원단체·국가인권위원회)

**C. 데이터 정직성**
- 검증 안 된 수치는 **`[추정]`** 명시
- 본문 마지막에 **데이터 정직성 선언** 필수
- 추정값과 공식 수치 명확히 구분

### 2.3 개발계획서.md

**A. 표 위주 작성** — 장문 서술 최소화. 표는 최대 12행 이내, 초과 시 분할.

**B. 필수 섹션**

| 섹션 | 내용 |
|---|---|
| 1. 기술 스택 | 계층별 표 (선택·버전·근거) |
| 2. 일정 (Gantt) | 시작일·종료일·현재상태 표 + Mermaid Gantt |
| 3. 마일스톤 | 날짜·산출물·달성여부 표 |
| 4. 스프린트 진척 | 스프린트별 상태 (✅ 완료 / 🚧 진행 / ⏸ 대기) |
| 5. 현재 상황 (Status) | 매 업데이트 시 갱신 |
| 6. 위험·이슈 | 발생일·영향·대응 표 |
| 7. 자원 사용 | API 비용·인프라 비용 표 |

**C. 현재 상황 갱신 의무**
- 코드 변경 / 빌드 / 배포 시 §5 갱신
- 갱신 시 `last_updated: YYYY-MM-DD HH:MM` 헤더 갱신

### 2.4 개발보고서.md

**A. 캡처 의무**
- **실제 구동된 화면을 스크린샷**으로 캡처
- 캡처는 `docs/captures/` 에 PNG로 저장
- 임베드: `![설명](./captures/파일명.png)`

**B. 작성 절차**
1. 코드 빌드·실행 → 실제 화면 띄움
2. 핵심 화면 캡처 (홈·핵심 기능·결과화면 최소 5장+)
3. 캡처와 함께 보고서 본문 작성
4. 캡처가 의도와 일치하는지 점검
5. 미흡하면 코드 수정 → 재구동 → 재캡처 → 보고서 재작성
6. 의도와 일치할 때까지 반복

**C. 검토 체크리스트 (보고서 마지막에 포함)**
- [ ] 모든 핵심 기능이 캡처되었는가
- [ ] 캡처가 의도한 기능을 정확히 보여주는가
- [ ] 한글이 깨지지 않는가
- [ ] 에러 화면이 의도치 않게 캡처되지 않았는가
- [ ] 결과물(PDF·텍스트)의 정확도가 충분한가

**D. 보고서 구조**

| 섹션 | 내용 |
|---|---|
| 1. 구현 범위 | 스코프 표 |
| 2. 환경 | OS·언어·라이브러리 버전 표 |
| 3. 실행 방법 | 재현 가능한 단계 |
| 4. 화면 캡처 | 캡처 + 설명 (각 화면당 1단락) |
| 5. 검증 결과 | 의도대로 작동하는지, 정확도 측정값 |
| 6. 미흡 사항 | 솔직한 기술 + 다음 액션 |
| 7. 검토 체크리스트 | 위 C 항목 |

## 3. 개발 규칙 (src/)

### 3.1 언어·도구
- **Python**: 3.11+ (3.14 사용 가능)
- **패키지 관리**: `pyproject.toml` + `pip` (또는 `uv`)
- **포맷터**: `ruff`
- **테스트**: `pytest`

### 3.2 환경변수 / 시크릿
- API 키는 **절대 코드에 하드코딩 금지**
- `.env` 파일 사용 + `.gitignore` 등록
- 키 부재 시 **모킹 모드**로 동작하게 설계 (데모 가능성 보장)

### 3.3 의존성
필요 시에만 추가, 미사용 의존성은 즉시 제거.

기본 스택:
- `streamlit` — 웹 데모 UI
- `openai` — Whisper STT
- LLM 클라이언트 (프로젝트 선택 기준)
- `reportlab` — PDF 생성
- `python-dotenv` — 환경변수
- `pydantic` — 데이터 검증

## 4. 코드 표준 (언어별)

준수 의무. 린터·포맷터 설정은 이 표준에 맞춰 둔다.

### 4.1 Python — **PEP 8 / PEP 257 / PEP 484**

| 범주 | 규칙 |
|---|---|
| 스타일 | PEP 8 (line length 100, `ruff` 기본 규칙) |
| 네이밍 | 함수·변수 `snake_case`, 클래스 `PascalCase`, 상수 `UPPER_SNAKE`, 모듈 `lower_snake` |
| Docstring | PEP 257 — 1줄 요약 + 필요 시 상세. 한국어 OK |
| 타입 힌트 | PEP 484 — public 함수·메서드는 인자·반환 타입 명시 |
| Import | 표준 라이브러리 → 서드파티 → 로컬 순서, 그룹 간 빈 줄 1개 |
| 주석 | WHY만 작성. WHAT은 코드가 설명하도록. 장식용 주석 금지 |
| 예외 | bare `except:` 금지. 구체적 예외만 잡는다 |
| 문자열 | f-string 우선, `%`/`.format()` 지양 |
| Path | `pathlib.Path` 우선, `os.path` 지양 |
| 로깅 | `logging` 모듈 사용, `print()` 남발 금지 |
| 테스트 | `pytest`, 함수명 `test_`, AAA 구조 (Arrange-Act-Assert) |

### 4.2 Markdown

| 범주 | 규칙 |
|---|---|
| 표준 | CommonMark + GFM 확장 (표·체크박스) |
| 제목 | `#`은 문서당 1개(제목), 이하 계층 구조 준수 |
| 줄바꿈 | 빈 줄로 단락 구분, 강제 개행(`<br>`) 최소화 |
| 코드블록 | 언어 태그 필수 (` ```python `) |
| 링크 | 상대 경로, 한국어 텍스트 링크 허용 |
| 표 | 헤더 가운데/좌/우 정렬 명시 |

### 4.3 Shell / Bash

| 범주 | 규칙 |
|---|---|
| Shebang | `#!/usr/bin/env bash` |
| 안전 모드 | `set -euo pipefail` 상단 고정 |
| 변수 | `"$var"` 큰따옴표 인용, `${var}` 중괄호 권장 |
| 린트 | `shellcheck` 통과 |

## 5. 작업 흐름

매 세션·큰 작업 단위마다:

```
1. 본 지침서 재확인
2. 작업 계획 수립 (TODO)
3. 코드/문서 변경
4. 코드 변경 시 → 실제 구동 → 캡처
5. docs/개발계획서.md §5 현재 상황 갱신
6. 의도와 다르면 → 수정 → 재구동 → 재캡처
7. docs/개발보고서.md 갱신 (캡처 포함)
8. TODO 완료 처리
```

## 6. 절대 하지 말 것

- ❌ 출처 없는 통계 인용
- ❌ 추정값을 사실로 표현
- ❌ 캡처 없이 "구현 완료" 보고
- ❌ API 키 하드코딩
- ❌ 검증 안 된 라이브러리 무분별 추가
- ❌ docs/ 외부에 문서 산재
- ❌ src/ 외부에 코드 산재

## 7. 우선순위 — Killer Feature First

콘테스트 기준 가치 順:

1. ⭐ **AI 파이프라인 PoC** (음성 → 한국어 산재신청서 PDF)
2. SOS 증거 수집 (사진·녹음·해시)
3. 전문가 매칭 (지도 + 다국어 검색)
4. 익명 커뮤니티 사례 DB
5. 권리 시효 알림

각 기능은 별도 디렉터리 + 자체 데모로 구동 가능해야 함.
